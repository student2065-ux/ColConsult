**Огляд проекту**

ColConsult, веб-система для організації запису на консультації між студентами та викладачами коледжу. Проєкт покликаний усунути хаос у плануванні, замінивши чати, соцмережі та паперові розклади на єдиний зручний інструмент.

:adhesive_bandage:**Проблема**

Студенти витрачають багато часу на пошук інформації про вільний час викладачів та узгодження консультацій через різні канали. Викладачі ведуть записи у зошитах або чатах, що призводить до плутанини, накладок та пропущених зустрічей. Адміністрації складно аналізувати навантаження викладачів через відсутність централізованих даних.

:wrench:**Pitch**

Система дозволяє:
Студентам: знаходити викладача та бронювати зручний час за 1 хвилину
Викладачам: керувати своїм розкладом консультацій та отримувати автоматичні сповіщення
Адміністрації: отримувати статистику навантаження для планування

Цільова аудиторія: студенти, викладачі та адміністрація коледжу.

:recycle:**MVP (Мінімально життєзданий продукт)**

Ключовий сценарій, без якого система не має сенсу:
Викладач створює вільний слот у своєму календарі
Студент переглядає список викладачів та вільний час
Студент обирає та бронює слот
Обидва отримують email-підтвердження

Роль	Очікування	Критерії успіху

Викладач-Контроль графіка, менше пропущених зустрічей.
Адміністрація-Статистика навантаження,	Можливість отримати звіт за місяць,
Технічний персонал,	Стабільна робота, легка підтримка	Мінімум запитів на підтримку.

:chart_with_upwards_trend:**Мета першої ітерації**

Запустити працюючу версію системи з основним сценарієм: викладач створює слот → студент бронює → обидва отримують підтвердження.

:hammer:**Критерії успіху**

Зручність: Студент завершує запис менш ніж за 10 кліків
Стабільність: Система працює тиждень без критичних збоїв (10+ користувачів)
Зрозумілість: 90% користувачів виконують основний сценарій без допомоги

:twisted_rightwards_arrows:**Детальний користувацький сценарій**

Умови початку:
Студент має обліковий запис та потребує консультації.

:children_crossing:**Кроки**

Вхід: Студент логіниться в систему

Пошук: Обирає факультет або вводить прізвище викладача

Перегляд: Натискає "Переглянути вільний час" на картці викладача

Вибір: Бачить календар (зелені — вільні слоти, сірі — зайняті)

Запис: Обирає дату/час → "Записатися"

Підтвердження: Перевіряє деталі → "Підтвердити"

Результат: Бачить "Запис успішний", отримує email


:fire:**Гіпотези та спостереження**


:memo:**Гіпотеза 1: Система зменшує час запису на консультацію"**

Основна цінність: Студент може записатися за 2 хвилини замість пошуку через чати (15+ хвилин)

Спостереження для перевірки:

Метрика часу: Відстежити середній час від відкриття системи до отримання підтвердження

Ціль: ≤ 2 хвилини для 80% користувачів

Інструмент: Google Analytics або власний логування

Завершення сценарію: Відсоток користувачів, які завершили запис

Ціль: ≥ 70% користувачів завершують процес

Перевірка: Аналіз логів кроків

Відгуки після використання: Збір коротких відгуків (1-2 речення)

Питання: "Наскільки простішим став запис порівняно з попереднім способом?"

Кількість: 20-30 відгуків від студентів

:memo:**Гіпотеза 2: Викладачі будуть оновлювати календар**

Основна цінність: Актуальний календар = можливість запису

Спостереження:

Активність викладачів: Кількість викладачів, які додали хоча б один слот

Ціль: ≥ 50% тестових викладачів

Період: перші 7 днів після запуску

Регулярність оновлень: Частота оновлення календарів

Ціль: 2+ оновлення на тиждень на викладача

Перевірка: Аналіз активності в БД

:passport_control:**Найголовніші невизначеності та як їх зменшити**

**1. Чи буде викладачам зручно додавати час?**

Ризик: Якщо складно — не будуть це робити

Спрощуємо в MVP:

Всі консультації по 1 годині

Можна швидко додати "кожен вівторок о 14:00"

Тільки 2 поля: дата і час

Важливо: Головне — щоб взагалі додавали

**2. Як сповіщати про запис?**

Ризик: Люди не побачать підтвердження

Спрощуємо в MVP:

Тільки email

Короткий лист: "Ви записані на..."

Плюс показувати підтвердження на екрані

Важливо: Навіть без email система працює

**3. Чи потрібен мобільний додаток?**

Ризик: Студенти користуються телефонами

Спрощуємо в MVP:

Зробити сайт під телефони

Можна додати на екран телефону

Перевірити на декількох телефонах

Важливо: Спочатку тестуємо веб-версію

**Що не міняємо:**

Основна ідея (запис на консультації)

Ключовий сценарій (знайти → вибрати → записатися)

Цінність (швидкий запис)


:necktie:**Definition of Success для першої ітерації**

**Ітерація вважається успішною, якщо виконано ВСІ умови:**

Кількісні метрики:
Конверсія: ≥ 70% студентів, які почали процес, завершили запис

Час: Середній час запису ≤ 2 хвилини

Активність: ≥ 50% тестових викладачів створили хоча б 1 слот

Стабільність: 0 критичних збоїв протягом 7 днів тестування

Якісні критерії:
Відгуки: ≥ 80% відгуків від студентів позитивні ("простіше, ніж було")

Зрозумілість: ≥ 90% тестових користувачів виконали сценарій без допомоги

Email доставка: ≥ 95% email-підтверджень доставлено протягом 5 хвилин

Технічні вимоги:
Продуктивність: Час відгуку API ≤ 1 секунда для 95% запитів

Доступність: Система доступна ≥ 99% часу робочого дня

Безпека: Всі дані передаються по HTTPS, паролі хешуються
